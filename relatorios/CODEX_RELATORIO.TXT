RELATORIO DE REVISAO DE CODIGO - SAA29
Data: 2026-03-28

Escopo revisado
- docs\requirements\00_SRS.md
- docs\requirements\01_SPECS.md
- modulos principais de autenticacao, panes, aeronaves e equipamentos

Objetivo
- Identificar bugs, defeitos, mas escolhas de implementacao, riscos arquiteturais e vulnerabilidades com foco em corretude, seguranca e aderencia aos requisitos.

======================================================================
1. RESUMO EXECUTIVO
======================================================================

Foram identificados problemas relevantes em controle de acesso, tratamento de uploads, rastreabilidade de conclusao de panes, aderencia parcial aos requisitos e semantica inconsistente de erros HTTP.

Os riscos mais importantes sao:
- criacao de usuarios sem autenticacao
- anexos expostos publicamente por rota estatica
- validacao fraca de upload baseada apenas em extensao
- conclusao de pane sem registrar corretamente o responsavel em todos os fluxos

Esses pontos afetam diretamente seguranca, integridade funcional e confiabilidade operacional do sistema.

======================================================================
2. ACHADOS
======================================================================

2.1 CRITICIDADE: ALTA

Item 1
Titulo: Criacao de usuarios sem autenticacao
Arquivos:
- app\auth\router.py

Descricao:
O endpoint POST /auth/usuarios nao exige usuario autenticado. Diferente de outros endpoints protegidos, ele nao recebe CurrentUser nem qualquer checagem de permissao.

Impacto:
- qualquer cliente que alcance a API pode criar uma conta valida
- quebra o requisito de acesso apenas apos autenticacao valida
- abre caminho para escalacao de acesso e abuso do sistema

Problema tecnico:
- a API de cadastro de usuario vira uma porta de entrada publica
- nao ha nenhuma restricao por perfil ou funcao

Correcao recomendada:
- exigir autenticacao no endpoint
- restringir criacao de usuarios a perfis autorizados, como INSPETOR ou ENCARREGADO
- se o sistema precisar bootstrap inicial, tratar esse caso separadamente e de forma controlada


Item 2
Titulo: Anexos expostos publicamente por arquivo estatico
Arquivos:
- app\main.py

Descricao:
O diretorio de uploads e montado diretamente em /uploads usando StaticFiles. Isso expõe os arquivos sem passar por autenticacao ou autorizacao.

Impacto:
- qualquer pessoa com a URL correta pode acessar evidencias, imagens ou documentos anexados
- vazamento de dados operacionais e de manutencao
- quebra do modelo de acesso autenticado esperado pelo sistema

Problema tecnico:
- arquivos deixam de obedecer as regras de autorizacao do dominio
- o controle passa a depender apenas do segredo do nome do arquivo

Correcao recomendada:
- remover exposicao publica via StaticFiles
- servir anexos por endpoint autenticado
- validar se o usuario tem permissao para acessar a pane correspondente antes de entregar o arquivo
- armazenar anexos fora de qualquer area servida diretamente pela aplicacao web


Item 3
Titulo: Validacao de upload fraca e fora da especificacao
Arquivos:
- app\panes\service.py
- docs\requirements\01_SPECS.md

Descricao:
O upload valida apenas a extensao do nome do arquivo. O parametro tipo_mime e recebido mas nao e usado. Alem disso, o sistema aceita .pdf, enquanto a especificacao fala explicitamente em jpg e png.

Impacto:
- um arquivo malicioso pode ser enviado com extensao permitida
- risco de armazenamento de payloads indevidos
- incoerencia entre implementacao e especificacao
- combinado com a exposicao publica de uploads, aumenta bastante a superficie de ataque

Problema tecnico:
- ausencia de validacao por MIME real e por assinatura do arquivo
- aceitacao por whitelist baseada em nome, o que e facilmente burlavel

Correcao recomendada:
- restringir os tipos aceitos conforme a especificacao, ou atualizar a especificacao formalmente se PDF for um requisito real
- validar content-type
- validar magic bytes ou assinatura do arquivo
- rejeitar inconsistencias entre extensao, MIME e assinatura
- registrar e monitorar tentativas invalidas


2.2 CRITICIDADE: MEDIA

Item 4
Titulo: Pane pode ser resolvida sem registrar o usuario que concluiu
Arquivos:
- app\panes\service.py
- app\panes\router.py

Descricao:
No fluxo de edicao via PUT /panes/{id}, a transicao para RESOLVIDA preenche data_conclusao, mas nao preenche concluido_por_id. Apenas o endpoint dedicado /concluir registra o responsavel.

Impacto:
- panes podem ficar concluídas sem rastreabilidade completa
- perda de auditoria sobre quem resolveu o problema
- descumprimento parcial da regra de rastreabilidade

Problema tecnico:
- dois caminhos diferentes levam ao mesmo estado final, mas com efeitos colaterais diferentes
- o estado RESOLVIDA pode ficar inconsistente dependendo da rota usada

Correcao recomendada:
- impedir que PUT /panes/{id} aceite status RESOLVIDA, forçando o uso de /concluir
ou
- passar o usuario autenticado para editar_pane e preencher concluido_por_id quando houver transicao para RESOLVIDA


Item 5
Titulo: Filtro por data existe na camada de servico, mas nao na API
Arquivos:
- app\panes\service.py
- app\panes\router.py
- docs\requirements\00_SRS.md
- docs\requirements\01_SPECS.md

Descricao:
O servico aceita data_inicio e data_fim, mas o endpoint de listagem de panes nao expõe esses parametros.

Impacto:
- requisito RF-06 fica parcialmente implementado
- a API nao entrega todas as capacidades previstas no documento

Problema tecnico:
- camada de servico e camada HTTP estao desalinhadas

Correcao recomendada:
- adicionar parametros data_inicio e data_fim no endpoint GET /panes/
- documentar o formato esperado
- incluir testes cobrindo os filtros por data


Item 6
Titulo: Montagem do diretorio estatico depende da ordem de inicializacao
Arquivos:
- app\main.py

Descricao:
O diretorio de uploads e criado no startup, mas o mount do StaticFiles ocorre antes, durante create_app. Se o diretorio nao existir na inicializacao, a rota estatica pode nao ser montada.

Impacto:
- comportamento inconsistente entre ambientes
- em deploy limpo, arquivos podem ser salvos mas nao servidos
- aumenta dificuldade de diagnostico

Problema tecnico:
- dependencia implícita de estado de filesystem antes da criacao da app

Correcao recomendada:
- criar o diretorio antes do app.mount
ou
- abandonar a rota estatica publica e servir anexos por endpoint autenticado


Item 7
Titulo: Mapeamento inconsistente de erros HTTP
Arquivos:
- app\aeronaves\router.py
- app\equipamentos\router.py
- app\panes\router.py

Descricao:
Diversos endpoints retornam 409 para recursos inexistentes ou 404 para conflitos de regra de negocio, dependendo do caminho.

Impacto:
- API menos previsivel para clientes
- integracao mais fragil
- dificulta tratamento automatizado de erros

Exemplos:
- associar controle a equipamento retorna 409 mesmo quando equipamento ou tipo de controle nao existem
- instalar item retorna 409 mesmo quando item ou aeronave nao existem
- remover instalacao retorna 404 para erro de regra de negocio e tambem para nao encontrado

Correcao recomendada:
- separar erros de nao encontrado, validacao e conflito de negocio
- padronizar:
  - 404 para entidade inexistente
  - 409 para conflito de estado ou regra de negocio
  - 422 para entrada invalida
  - 400 apenas para casos realmente genericos


2.3 CRITICIDADE: BAIXA A MEDIA

Item 8
Titulo: Commit automatico dentro da dependencia de banco mascara fronteiras transacionais
Arquivos:
- app\dependencies.py

Descricao:
A dependencia get_db faz commit automaticamente ao final da requisicao. Isso simplifica o uso, mas mistura controle transacional com injecao de dependencia.

Impacto:
- dificulta controle fino de transacoes
- pode complicar cenarios futuros com multiplas operacoes que precisem de rollback mais explicito
- torna os servicos menos explicitos sobre persistencia

Problema tecnico:
- a fronteira de commit fica invisivel para quem chama os servicos

Correcao recomendada:
- considerar mover a responsabilidade transacional para camada mais explicita
- se mantiver o padrao atual, documentar claramente e manter consistencia em todos os endpoints


Item 9
Titulo: Documentacao e comportamento estao desalinhados em alguns pontos
Arquivos:
- app\panes\models.py
- app\panes\service.py
- docs\requirements\00_SRS.md
- docs\requirements\01_SPECS.md

Descricao:
Ha comentarios e contratos que nao batem completamente com o comportamento real.

Exemplos:
- o comentario de RN-03 fala em apenas panes abertas, mas a implementacao aceita ABERTA e EM_PESQUISA
- a especificacao de upload fala em jpg e png, enquanto o codigo aceita pdf
- o endpoint de conclusao recebe observacao_conclusao, mas essa informacao nao e persistida

Impacto:
- confusao de manutencao
- risco de teste enganoso
- dificuldade de evolucao do produto

Correcao recomendada:
- alinhar especificacao, comentarios e codigo
- remover campos de API sem efeito persistente
ou
- implementar persistencia se a observacao de conclusao for realmente necessaria


Item 10
Titulo: Cobertura de testes nao protege alguns cenarios criticos
Arquivos:
- tests\test_auth.py
- tests\test_panes.py
- tests\test_equipamentos.py

Descricao:
Os testes existentes cobrem fluxos principais, mas nao cobrem parte dos riscos mais importantes encontrados nesta revisao.

Lacunas observadas:
- ausencia de teste que prove que POST /auth/usuarios deve exigir autenticacao
- ausencia de teste cobrindo acesso indevido a anexos
- ausencia de teste que garanta concluido_por_id ao resolver uma pane
- ausencia de teste para filtros por data na API
- ausencia de teste de validacao real de tipo de arquivo

Impacto:
- regressões de seguranca e corretude podem passar despercebidas

Correcao recomendada:
- adicionar testes de autorizacao, de acesso a anexos, de rastreabilidade e de validacao binaria de upload

======================================================================
3. MAS ESCOLHAS DE IMPLEMENTACAO
======================================================================

3.1 Exposicao publica de uploads
- Escolha inadequada para um sistema com autenticacao obrigatoria e dados operacionais sensiveis.

3.2 Validacao por extensao de arquivo
- E uma estrategia fraca do ponto de vista de seguranca.

3.3 Regras de negocio distribuidas em fluxos com efeitos diferentes
- Resolver pane por PUT e por POST /concluir gera estados finais distintos quanto a auditoria.

3.4 Tratamento generico de ValueError nos routers
- Reduz qualidade semantica da API e mistura casos de inexistencia com conflito.

3.5 Contratos de API parcialmente desconectados dos requisitos
- Filtro de data existe no servico, mas nao no endpoint.
- Campo observacao_conclusao existe na API, mas nao participa da persistencia.

======================================================================
4. VULNERABILIDADES E RISCOS DE SEGURANCA
======================================================================

Risco 1
Tipo: Controle de acesso inadequado
Descricao:
- criacao de contas sem autenticacao

Risco 2
Tipo: Exposicao indevida de arquivos
Descricao:
- anexos acessiveis fora da camada de autorizacao

Risco 3
Tipo: Upload insuficientemente validado
Descricao:
- validacao apenas por extensao
- MIME ignorado
- assinatura do arquivo nao validada

Risco 4
Tipo: Rastreabilidade incompleta
Descricao:
- conclusao sem conclusor em um dos fluxos possiveis

5. PRIORIZACAO DE CORRECOES
======================================================================

Prioridade imediata

2. Remover exposicao publica de uploads
3. Endurecer validacao de uploads
4. Unificar o fluxo de conclusao de pane para sempre registrar concluido_por_id

Prioridade seguinte
5. Expor filtro por data na API de panes
6. Corrigir mapeamento de erros HTTP
7. Revisar contratos da API versus especificacao
8. Reforcar testes para cenarios de seguranca e auditoria
